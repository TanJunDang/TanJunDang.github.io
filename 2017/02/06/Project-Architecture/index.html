Android项目架构模式

MVC模式

理解
1.MVC分别指的的模型、视图、控制器。
2.在Android中，Model可以理解为业务逻辑处理(一些实体类、一些请求封装等)，View就是控件，控制器可以理解为控制Data如何在View上显示。

MVP模式

MVP模式的优点
使代码更加灵活、模块独立、单元测试方便、降低耦合度。这里的M、V、P都是一个接口。
MVP模式的缺点
使Presenter的代码量过多，接口过多，一定程度影响了编码效率。

对MVP的理解
M指模型接口，主要负责请求并获取数据，回传给Presenter。
V是视图接口，由ACTIVITY、FRAGMENT实现响应用户操作交给P处理。
P是逻辑处理接口，拥有M、V两者的引用，负责业务逻辑处理(从M获取数据，返回给P，P再传给V更新)。

MVP中M、V、P的特点分别是什么？
M不与V直接联系，需要P作为中间者继续数据传递。

通用的MVP架构
定义View接口
1.提供更新View的方法。
2.持有Presenter的引用，提供回调方法。（回调更新成功不是必要的）
定义Model接口
1.定义请求数据的方法。
定义Presenter接口
1.持有View、Model的引用。
2.添加model处理成功/处理失败的方法。
Activity的处理
持有Presenter引用，实现View接口。
